## 2. Workflow Gaps Found

### 2.1 No push/PR skill — commit and PR are separate but unguided
**Step in workflow:** done (after commit)
**Current state:** The `update-docs-and-commit` skill ends at commit with an explicit "No push" rule. There is no `/push` or `/pr` command to continue from there. After committing, the developer must remember to push the branch and create a PR manually, following the branch-naming convention and PR description requirements. The deleted `commit-push.md` command is gone and nothing replaces its push/PR half.
**Impact:** Medium — happens after every phase (roughly once per major feature). The PR requirements live in `workflow-gates.md` but there is no skill or command to guide executing them. Developers must recall the `gh pr create` format and branch naming rule from memory.
**Proposed fix:** Create a `/push-pr` command and companion skill (`push-and-pr`) that runs `git push`, then invokes `gh pr create` with the standard PR body template (what/why/testing notes), and verifies branch naming matches `feat/phase-NN-<short-name>` or `fix/<component>-<issue>`.

### 2.2 No supplier onboarding skill — first-scan for unknown supplier is ad-hoc
**Step in workflow:** code / debug
**Current state:** When a scan arrives for a supplier not yet in `backend/data/suppliers/`, the engine creates a new directory via `normalize_supplier_id()`. There is no guided workflow for this. The developer must manually create a profile, add a layout descriptor, and verify the first scan — all steps that require knowing the data schema. This is a blank spot in both skills and commands.
**Impact:** High — Phase 22's changelog explicitly lists adding 6 real supplier profiles as a significant milestone. Each new supplier will repeat this friction.
**Proposed fix:** Create a `new-supplier` skill that walks through: create slug, create `profile.json` with correct schema, run a test scan with `/scan`, verify memory update, and create a layout profile. This is documented work that happens multiple times and always follows the same steps.

## 3. New Skills / Agents Proposed

### 3.1 `new-supplier` skill
**Type:** skill
**Trigger:** When a user wants to add a new restaurant supplier to the scanner memory, or when a scan arrives for an unknown supplier and the developer wants to create a proper profile.
**What it does:** Guides creation of a new supplier directory under `backend/data/suppliers/<slug>/`. Walks through: generating the normalized slug via `normalize_supplier_id()`, creating `profile.json` with the required schema (supplier name, common fields, confidence baselines), running an initial scan with `/scan` to populate memory, and verifying the profile was written correctly. Prompts the developer to add a layout descriptor if the supplier has a consistent invoice layout.
**Why it's worth creating:** Phase 22 added 6 supplier profiles and this was a significant enough effort to appear in the changelog. Every new restaurant client onboarded will require this. The current skill set has no guidance for it, making each onboarding ad-hoc.
**Priority:** High

### 3.2 `push-and-pr` skill + `/push-pr` command
**Type:** skill + command
**Trigger:** After the `update-docs-and-commit` workflow completes and the user is ready to push to remote and open a PR.
**What it does:** Runs `git push -u origin <current-branch>`, verifies branch name matches `feat/phase-NN-<short-name>` or `fix/<component>-<issue>` convention, then constructs a `gh pr create` call with the standard PR body template (What changed / Why / Testing notes). Enforces the rule that PRs never target `master` directly from an unreviewed branch.
**Why it's worth creating:** The `update-docs-and-commit` skill explicitly stops at commit — it never pushes. But after committing, the developer still needs to push and open a PR, following PR conventions defined in `workflow-gates.md`. There is currently no command for this second half. Every phase ends with this step and it is currently ad-hoc.
**Priority:** Medium
